02 — The IDX website

Real MLS data without a search API

MLS Grid is a replication feed, not a search API — you may only filter on a handful of system fields, never on county, price, or beds. That single constraint shaped the whole data design.

The constraint that drove the design MLS Grid v2 $filter is limited to ListingId, StandardStatus, ModificationTimestamp, PropertyType, OriginatingSystemName, MlgCanView. You cannot ask it for "3-bed homes in Dutchess under $600k." So listing data is replicated into a local snapshot and filtered in-app; listing photos are fetched live, per view, and never stored.

Search & filters

County, price, beds, baths, sqft, property type, text, and sort — all computed in-memory over the snapshot, so a query never touches MLS on the request path.

Zillow-style map

Leaflet with price pins. The feed carries no lat/long, so pins are zip-centroid approximations with an honest "locations approximate" badge; the payload is coordinate-trimmed.

On-demand listing pages

One dynamic /listing/[id] template resolves any listing on request — no pre-built pages, no stored photos, compliant MLS attribution on every surface.

03 — The CRM

Brivity parity, plus an engine that actually runs

A dark-theme CRM shaped after Brivity — people, transactions, reporting, CMA — on Supabase Postgres with row-level security on every table. The headline is the Automations engine: not a plan builder, a plan runner.

Data model

23 tables — organizations, contacts, tasks, listings, transactions, activities, auto-plans & steps, lead ponds, CMA reports — with row-level security enabled on every one. Access is org-scoped through Supabase SSR sessions.

Compliance built in

Outbound marketing SMS carries a "Reply STOP to opt out" footer; a suppression list is checked before every send; an in-CRM opt-out control writes to it. Inbound-STOP webhook + A2P registration are the owner's go-live steps.

Why route sends through n8n

n8n already holds the Twilio and Gmail credentials and is a real automation backend. The CRM calls one dispatch webhook rather than embedding provider SDKs — fewer secrets in the app, one place to observe every send.

Safe by default

With no dispatch env configured the engine logs test runs and delivers nothing. Turning on real sends is a deliberate, single, owner-gated switch — not an accident waiting to happen.

04 — The AI recruiter page

A 3D journey that converts on a phone

A scroll-driven WebGL story — a galaxy that flies in and reshapes into a neural brain, then opens into the services hub — with a live assistant and a recruit form. Most visitors are on phones, so mobile is the priority, not an afterthought.

Mobile-first WebGL

Device-pixel-ratio scaling, a frame-rate governor, portrait camera framing so the brain reads as a brain, safe-area/notch handling, and 16px form inputs so iOS never zoom-jumps the canvas.

Boot resilience

A watchdog shows a static "here's the offer + Work with me" fallback if the 3D app can't start (an old browser, a failed asset, a lost GL context) — the conversion path survives even when WebGL doesn't.

Served under one domain

Its own Vercel project, CLI-deployed, proxied at /ai so it inherits the marketing site's strict CSP and noindex — one brand, one security posture.

05 — The autonomous agent loop

How a solo operator ships three apps

Much of this was built by a supervised agent loop: an orchestrator scopes work, parallel sub-agents execute per app, every claim is verified against the live system before it's accepted, and a watchdog guards the process itself.

  • Split by property, not one mega-agent. Different stacks run in parallel and stay deep instead of one context going shallow.
  • Verify independently. A sub-agent's "done" is never trusted — the orchestrator re-checks it against the deployed URL, the database, or a live probe.
  • Safety by construction. One writer per repo, the live CRM loop untouched, a STOP file and rogue-commit triggers, and secrets never committed (verified across full git history).

07 — Security & deployment

Locked down before it's public

The whole suite is private and non-indexed until launch, and the money paths are hardened by construction.

Headers & CSP

One unified Content-Security-Policy across the site and the proxied 3D app, plus HSTS-preload, nosniff, frame-DENY, Referrer-Policy and Permissions-Policy — verified 0 violations live.

The photo proxy can't be abused

The listing id is regex-validated and only ever expands into MLS's own signed URLs — no attacker-controlled URL ever reaches a fetch. No SSRF surface.

Lead intake is defended

Content-type enforced, body size-capped, a honeypot, and a per-IP rate limit that returns 429 — with Vercel WAF as the durable edge layer at launch.

Secrets & data

Zero secrets in any repo's git history (scanned). All keys live in Vercel/Supabase env. RLS on every CRM table. Automations default to a no-send test mode.
